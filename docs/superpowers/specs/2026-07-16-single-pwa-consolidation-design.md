# POLSKA 單一 PWA 系統整併設計

日期：2026-07-16（Asia/Taipei）

## 結論

POLSKA 將整併為一個正式產品：以現有 `B-companion` 手機旅伴為核心的響應式 PWA。根網址 `/`、安裝後啟動頁與手機／桌機瀏覽都進入同一套應用，不再依裝置跳轉至不同系統。

手機是主要使用情境；桌機不是另一個版本，而是同一套 PWA 在寬螢幕下的閱讀排版。經典版、桌機雜誌版與 App 預覽版第一階段不刪除，只退出正式導覽、PWA 快取及日常維護路徑，避免不可逆處理。

## 使用者會得到什麼

- 只需記住並開啟一個網址。
- 加到 iPhone 主畫面後，直接進入「今日」旅伴。
- 即使網路不穩，仍可查看核心行程、交通、訂票及自己的本機備註。
- 手機底部固定保留「今日／行程／交通／訂票」四個主要入口。
- 電腦開啟同一網址時，內容自動改為較寬、較容易閱讀的版面，但功能與資料完全相同。
- 更新後有清楚提示，不再因多套快取版本而長期看到舊畫面。

## 現況問題

目前 `/index.html` 會依裝置跳往 `mobile.html` 或 `desktop.html`，另可用 `?classic=1` 停留在手寫經典版；`app-preview.html` 又提供另一個展示入口。程式層同時存在 A、B、C 三套介面，Service Worker 也會預先快取經典版、桌機版與預覽版。

雖然 `redesign/data.js` 已是 React 介面的共用資料來源，但經典版仍包含大量手寫行程內容，形成第二條內容維護路徑。這會讓同一筆行程更新需要同步多處，也提高手機收到舊快取與各版本內容不一致的風險。

## 方案比較

### 方案 A：保留所有版本，只統一資料

優點是外觀與既有網址幾乎不變；缺點是多套 UI、跳轉規則、編譯產物及快取仍需維護，無法真正解決版本混亂。

### 方案 B：全站只保留固定寬度手機畫面

優點是改動量最小、PWA 定位最明確；缺點是桌機閱讀浪費螢幕空間，也不利於行前規劃與一次比較多天行程。

### 方案 C：單一 PWA，加上響應式桌機排版（採用）

以 `B-companion` 為唯一功能核心，手機維持 App 式單欄操作，寬螢幕改為雙欄或主內容加輔助資訊。所有裝置共用相同狀態、資料、導覽及快取。此方案能真正消除版本分叉，同時保留桌機可讀性。

## 產品架構

### 唯一正式入口

- `/` 是公開、分享與書籤使用的唯一正式入口。
- `manifest.json` 的 `id`、`start_url` 與 shortcuts 均指向 `/` 或同一套應用內的定位方式。
- 不再用 User-Agent 或寬度判斷後跳頁。
- `mobile.html`、`desktop.html`、`app-preview.html` 與 `?classic=1` 第一階段改為相容轉址或清楚的舊入口提示，不再呈現獨立功能版本。

### 唯一介面核心

- 以 `redesign/B-companion.jsx` 為主體，依職責拆成可獨立理解與驗證的區塊。
- 四個底部入口維持「今日／行程／交通／訂票」。
- 手機使用固定底部導覽與單欄內容；桌機保留相同四個入口，但可採側欄或較寬的內容排列。
- `A-magazine` 的有價值資訊，例如行前總覽、城市故事與完整閱讀視角，整合回單一應用中的次要閱讀區，不保留另一套應用殼。
- `C-app` 與 iOS frame 只屬設計研究，不再參與正式 build。

### 單一資料來源

- `redesign/data.js` 維持旅程資料的唯一程式來源。
- 所有日次、交通、訂票、備案、城市故事與餐食資訊均由同一資料物件產生。
- README 只說明使用方式與維護方式，不再複製完整行程表作為第二份內容來源。
- 使用者備註繼續保存在裝置的 `localStorage`；本階段不新增帳號、雲端同步或後端。

## 手機 PWA 強化範圍

### 安裝與啟動

- iPhone Safari 一般瀏覽時提供不干擾操作的「加入主畫面」說明；安裝後不再顯示。
- standalone 模式正確處理瀏海、狀態列及底部安全區。
- 點開 PWA 時直接回到今日頁，並依旅程日期自動選擇 Day 1–8；旅程日期外提供清楚的預覽狀態。

### 旅途中高頻操作

- 首屏優先顯示現在行程、下一個硬時間、下一站與離線狀態。
- 地點與車站可一鍵開啟地圖；離線時保留地址／站名等可抄寫資訊，並明確告知外部地圖需要網路。
- 訂票頁集中顯示必訂項目、官方連結與建議期限，不加入虛假的已購票狀態。
- 個人備註可在各行程步驟新增與修改，資料只留在該裝置。

### 離線與更新

- Service Worker 只預先快取單一應用所需的 HTML、CSS、JavaScript、圖示與行程資料。
- 導覽採 network-first 並以單一應用殼作為離線 fallback；靜態資源採可控版本的 cache-first。
- 安裝失敗不得留下「看似可離線但內容不完整」的狀態；核心資源缺漏時應讓 install 失敗，保留上一個可用版本。
- 新版本準備完成後顯示更新提示，由使用者點擊後切換並重新載入；避免旅行途中在操作中途自動刷新。

## 錯誤與降級處理

- 無網路：顯示離線狀態，但核心行程仍可操作。
- 地圖／訂票外部連結不可用：顯示需要連線的白話提示，不讓按鈕無反應。
- `localStorage` 不可用：備註功能降級為當次使用，並提示未能永久保存。
- Service Worker 不支援或註冊失敗：網站仍能正常線上使用，PWA 狀態區說明離線功能未啟用。
- 舊入口：保留相容導向，避免既有書籤出現 404；不讓舊版繼續形成內容分支。

## 不在本次範圍

- 不新增登入、多人共用、雲端同步、後端或資料庫。
- 不新增定位追蹤、即時火車 API、推播通知或付款功能。
- 不改變已確認的 8 天 7 晚行程內容。
- 未經使用者另行確認，不刪除舊版原始檔。
- 本次不部署；完成 build、test、手機與桌機實際渲染、PWA 離線驗收及交叉審查後，再另行詢問是否對外發布。

## 驗收條件

1. 手機開啟根網址後不再跳轉到另一份 HTML，直接看到 POLSKA PWA。
2. iPhone 尺寸下底部四個入口固定可用，頁面沒有橫向溢出，瀏海與 Home Indicator 不遮內容。
3. 桌機開啟同一根網址，可舒適閱讀同一套資料與功能，不顯示另一套雜誌應用。
4. 安裝後從主畫面啟動，進入同一根網址與 standalone 介面。
5. 完成一次線上載入後切斷網路，重新開啟仍能查看 Day 1–8、交通與訂票核心資料。
6. 新版本可顯示更新提示；使用者確認後切換，舊快取會被清理。
7. 舊的 `mobile.html`、`desktop.html` 與 `app-preview.html` 網址不出現 404，會安全回到單一 PWA。
8. build、靜態檢查與自動測試全數通過，並以實際瀏覽器驗證 320 px、390 px 與桌機寬度。
9. 不提交 `.env`、憑證或其他 secrets，不動正式資料。

## 測試策略

- 對日期選日、離線降級、舊入口導向與備註保存補上可重複的邏輯測試。
- 執行 `./build.sh`，確認正式使用的 JSX 能產生可載入的 dist 檔案。
- 啟動本機 HTTP server，以瀏覽器檢查根網址、舊入口與 Service Worker；不使用 `file://` 驗證 PWA。
- 在 320 px、390 px 與桌機 viewport 驗證主要導覽、無水平溢出、固定底欄與焦點操作。
- 先上線載入，再模擬離線重新整理，確認核心內容與更新流程。
- Codex 實作完成後，依 `DEVELOPMENT.md` 交由 Claude 進行獨立審查；使用者最後實際操作一次核心路徑。

## 遷移順序

1. 建立單一根入口與相容導向，不改旅程內容。
2. 將 B 旅伴調整為同時支援手機與桌機的唯一應用。
3. 收斂 manifest、Service Worker、build 與 README。
4. 補齊 PWA 安裝、離線失敗、外部連結及備註降級提示。
5. 完成自動與真實瀏覽器驗收，再交叉審查。
6. 使用者確認後，才決定是否移除已退出使用的舊版檔案及是否部署。
